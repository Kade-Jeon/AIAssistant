주요 프로젝트 / 성과
상황 레거시 메일 서비스에는 이미 AI 호출 기능이 있었는데, RestTemplate으로 LLM API를 직접 호출하는 방식이었습니다. 시스템 프롬프트·사용자 메시지·컨텍스트가 구분 없이 하나의 문자열로 합쳐져 전송되다 보니 AI 응답 품질이 낮았고, 프롬프트 엔지니어링을 반복해도 응답 품질이 오르지 않았으며 RAG·에이전트 구조로 확장하는 것도 구조적으로 불가능한 상태였습니다.
행동 회사 내 Java/Spring 기술 스택에 대한 선호가 강한 점을 감안해, Python 계열 프레임워크(LangChain 등) 대신 Spring AI 도입을 직접 제안했습니다. Spring AI의 ChatClient 구조로 시스템 프롬프트·메시지 역할을 명확히 분리하고, 1차로 메일 본문 번역·외국어 메일 작성 기능을 구현했습니다. 동시에 이후 RAG 파이프라인, 메일·주소록 도메인 단위 에이전트 구조, Langfuse 기반 Observability로 확장할 수 있도록 아키텍처를 설계했습니다. 도입 제안부터 설계·구현까지 대부분을 담당했습니다.
결과 한국지능정보사회진흥원(NIA) 대상 번역·메일 작성 AI 기능 베타 운영 도입 (2026.01~). 단일 텍스트로 합쳐 전송하던 구조적 한계로 인해 프롬프트 엔지니어링을 반복해도 응답 품질이 오르지 않던 문제를 해소했고, ChatClient 역할 분리 구조 전환 후 응답 일관성이 눈에 띄게 향상되었습니다. 공공 클라우드 서비스 최초 AI 기능으로 채택되었고, 이후 RAG 기능 추가 확장 요청이 들어왔지만 사전 설계 덕분에 재설계 없이 대응 가능한 상태입니다.
배운 점 아무리 프롬프트 엔지니어링을 반복해도 응답 품질이 오르지 않는 상황에서, 문제의 원인이 프롬프트 내용이 아니라 구조 자체에 있었음을 파악했습니다. 기능을 동작하게 만드는 것과 확장 가능하게 설계하는 것은 다른 문제임을 실감했고, 사전에 확장을 고려한 설계 덕분에 이후 RAG 기능 추가 요청에 재설계 없이 대응할 수 있었습니다. 기능 도입 시 확장 가능성까지 함께 설계하는 것이 실질적인 유지보수 비용을 줄인다는 것을 이 프로젝트에서 직접 확인했습니다.
상황 기존에는 Scouter로 일부 메트릭만 확인 가능했고, 로그를 보려면 관련 인원이 모든 VM에 root로 접속해 분산된 로그를 직접 확인해야 해서 장애 원인 파악이 느리고 보안·운영 리스크가 있는 상태였습니다.
행동 Scouter 대비 로그·트랜잭션·슬로우쿼리까지 한 곳에서 분석 가능하도록 Docker 기반 ELK Stack(Elasticsearch·Logstash·Kibana)을 구성하고, Fleet + Elastic Agent로 중앙 수집, Admin/Readonly 권한 분리, Elastic APM으로 Java 메일 서비스의 성능·트랜잭션 모니터링을 구성했습니다.
결과 한국과학기술정보연구원(KISTI) 프라이빗 클라우드에 도입 (2025.02~). VM root 접속을 필수 인원으로 제한하고 그 외 인원은 Kibana를 통해 로그를 조회하도록 전환해 보안을 강화했고, 중앙 집중 조회로 장애·CS 대응 시간을 기존 대비 약 30% 수준으로 단축했습니다.

또한 Elastic APM 슬로우쿼리 로그에서 수신확인 기능의 메일 조회 SQL이 반복 감지되는 것을 발견했습니다. 보낸 메일 전용 기능임에도 전체 메일함 조회 로직을 그대로 재사용해 is_spam = 0·is_deleted = 0 조건이 남아 있었고, selectivity가 없는 필터로 작동하고 있었습니다. 해당 조건을 제거해 쿼리 성능을 개선했고, 기존에는 각 서버 로그만 봐서는 발견하지 못했던 병목을 ELK 대시보드에서 직접 찾아낼 수 있었습니다.
배운 점 다중 서버를 직접 오가며 로그를 보는 방식의 비효율을 체감하고, 도입 근거와 구성안을 직접 정리해 제안했습니다. 모니터링 가시성이 생기자 기존에는 누구도 인지하지 못했던 슬로우쿼리가 보이기 시작했고, 이를 발견해 직접 개선할 수 있었습니다. 문제는 보여야 해결할 수 있다는 것, 그리고 불편함에서 끝내지 않고 해결 방향을 제시하는 것이 개발자의 역할임을 확인했습니다.
상황 클라우드 메일 서비스 고객사에서 기존 SMS 기능 외에 MMS(이미지 첨부 발송) 기능 추가를 요청했습니다. 기존 SMS 로직이 확장을 고려하지 않은 구조여서, 그대로 MMS를 얹으면 기존 기능 안정성에 영향을 줄 우려가 있었습니다.
행동 기존 SMS가 shell curl 기반으로 통신사 API를 호출하고 있어, 통신사 최신 API 문서를 기반으로 RestTemplate 방식으로 재구현했습니다. 공통 발송 로직과 타입별 처리 로직을 분리해 재설계하고, 통신사 MMS API 연동·이미지 파일 검증·예외 처리를 단독으로 담당했습니다. 엑셀 대량 발송 시 통신사 1회 요청 수신자 제한을 고려해 청크 단위로 분할하고, CompletableFuture로 병렬 발송 후 전체 결과를 집계해 결과 페이지에 반환하도록 구현했습니다.
결과 한국농수산식품유통공사 운영에 적용되었습니다 (2025.06~). 엑셀 기반 대량 발송 도입으로 기존 건별 수동 등록 방식 대비 발송 준비 시간이 크게 단축되었고, SMS 단독 운영에서 MMS까지 발송 가능한 콘텐츠 유형이 확대되었습니다.
배운 점 shell curl로 작성된 레거시 SMS 코드를 RestTemplate으로 재구현하면서, 기존 코드가 왜 그렇게 작성됐는지 파악하는 것부터 시작해야 했습니다. 통신사 API의 수신자 제한이라는 외부 제약을 설계에 반영해 청크 분할·병렬 처리 구조로 대응했고, API 문서가 불충분한 상황에서 직접 문의하며 완성한 경험을 통해 스펙 출처·제약·참고 링크를 코드에 명시하는 문서화 원칙을 세우게 되었습니다.

② AgentToolProvider가 요청별 userId+projectId 컨텍스트를 생성자 주입으로 전달 → ThreadLocal 없이 Virtual Thread 환경에서 안전한 컨텍스트 격리
③ PromptType.PROJECT일 때만 RAG 툴 활성화로 관심사 분리
④ Tool 구조로 에이전트를 캡슐화하면 추후 독립 서비스로 분리하여 메일 외 다른 도메인(챗봇 등)에도 재사용 가능한 확장성 확보
맥락 SSE 스트리밍 환경에서 X-Idempotency-Key의 상태(없음/진행 중/완료/실패)에 따라 처리 로직이 완전히 달라집니다. 결정 IdempotencyStateHandler 인터페이스 + 4가지 구현체: NoStateHandler, InProgressStateHandler, CompletedStateHandler, FailedStateHandler 근거 ① 각 상태 처리 로직을 독립 클래스로 캡슐화 → SRP
② 새 상태 추가 시 기존 코드 수정 없이 구현체 추가 → OCP
③ StreamingIdempotencyCoordinator가 상태 판별 후 핸들러에 위임하는 구조로 관심사 분리
맥락 매 AI 요청마다 이전 대화 20개를 조회해야 합니다. DB만 사용하면 지연이 발생하고, Redis만 사용하면 서버 재시작 시 대화 맥락이 소실됩니다. 결정 Redis를 L1 캐시(RedisChatMemory), PostgreSQL을 L2 영구 저장소(CustomChatMemoryRepository)로 사용하는 이중 레이어 구조 근거 ① 응답 지연 최소화 + 데이터 내구성 동시 확보
② Spring AI ChatMemoryRepository 인터페이스 직접 구현으로 중복 메시지 방지 로직 포함
③ JdbcChatMemoryRepositoryAutoConfiguration 제외 후 커스텀 구현체 사용
맥락 SSE 스트리밍으로 다수의 동시 연결이 발생하며, 각 연결은 Ollama 응답을 기다리는 I/O 블로킹 작업을 포함합니다. 결정 Java 21 Virtual Thread 전면 활성화 (spring.threads.virtual.enabled: true) + SSE 전용 Scheduler 분리 근거 ① 기존 플랫폼 스레드 풀은 스레드 수 제한으로 동시 SSE 연결에 병목
② Virtual Thread는 I/O 블로킹 시 캐리어 스레드를 해제 → 수천 개 동시 연결 처리 가능
③ SseStreamingConfig로 SSE I/O를 메인 스레드와 분리
맥락 AI 모델의 동작(프롬프트, temperature, topK 등)을 변경할 때마다 코드 배포가 필요하면 운영 유연성이 낮습니다. 결정 Langfuse에 프롬프트 템플릿과 모델 파라미터를 저장 → 런타임에 API 조회 → Redis 1시간 TTL 캐시로 성능 최적화 근거 ① 코드 배포 없이 temperature, topK, topP, 시스템 프롬프트 즉시 변경 가능
② LangfuseClient에 5xx 재시도(최대 3회, delay 2초) 내장
③ LangfusePromptTemplate에 모델 파라미터까지 포함 → 프롬프트와 모델 설정 일원화 관리
🐛 트러블슈팅
문제 Spring AI 기본 JdbcChatMemoryRepository는 메시지 저장 시 기존 대화 전체를 삭제 후 재삽입하는 방식으로 동작합니다. 요청마다 CHAT_MESSAGE 테이블의 레코드 ID가 재생성되어, 첨부파일 정보를 저장하는 CHAT_ATTACHMENT와 FK 연관관계를 맺는 것이 불가능했습니다. 해결 ChatMemoryRepository 인터페이스를 직접 구현한 CustomChatMemoryRepository를 작성하여 기본 자동 구성(JdbcChatMemoryRepositoryAutoConfiguration)을 비활성화하고 교체했습니다.

saveAll()을 전체 교체(delete + insert) → 추가(append-only) 방식으로 재정의. 저장 전 type + content 기준으로 중복을 검사해 신규 메시지만 INSERT하고, ChatMessageEntity 생성 시 UUID.randomUUID()로 ID를 최초 1회만 할당하여 이후 변경되지 않는 구조를 만들었습니다. 결과 CHAT_MESSAGE ID가 안정적으로 유지되어 CHAT_ATTACHMENT와의 FK 연관관계 및 JPA @OneToMany 매핑이 정상 동작. 프레임워크의 자동 구성보다 인터페이스 직접 구현이 도메인 요구사항을 더 정밀하게 제어할 수 있음을 확인. OCP 원칙에 따라 기존 코드 수정 없이 구현체 교체만으로 문제 해결.
문제 SSE 스트리밍 환경은 네트워크 연결이 불안정할 수 있습니다. 클라이언트가 재시도를 수행할 때, 서버에서 별도의 처리가 없으면 동일한 사용자 메시지가 DB에 중복 저장되고, 고비용의 AI 모델 호출이 중복 발생하여 자원이 낭비되는 문제가 있었습니다. 해결 X-Idempotency-Key를 도입하고 Redis를 활용해 요청 상태(IN_PROGRESS, COMPLETED, FAILED)를 관리하는 로직을 구현했습니다.

재시도 요청 시 상태가 FAILED라면, 기존에 할당된 conversationId와 userMessageId를 조회하여 재사용합니다. 이를 통해 사용자 메시지 저장 로직을 스킵하고 AI 모델 호출만 다시 수행하도록 설계했습니다. 결과 네트워크 장애로 인한 재시도 상황에서도 DB 데이터의 무결성을 100% 보장하게 되었습니다. 또한 불필요한 중복 AI 추론을 방지하여 서버 리소스를 최적화하고, 사용자에게는 끊김 없는 대화 경험을 제공할 수 있게 되었습니다.

💡 주요 기술적 의사결정
결정 모든 API 호출을 useApi.ts 단일 진입점으로 집중, 내부에서 Fetch API 직접 사용 근거 ① EventSource 사용 불가 — 브라우저 내장 SSE 클라이언트인 EventSource는 GET 요청만 지원합니다. 이 프로젝트는 /api/v1/ai/conv에 POST로 메시지를 전송하면서 SSE 스트림을 응답으로 받는 구조라 사용이 불가능했습니다.

② axios의 브라우저 스트리밍 한계 — axios는 브라우저 환경에서 내부적으로 XHR을 사용하는데, XHR은 response.body.getReader()를 지원하지 않습니다. XHR의 onDownloadProgress로 일부 구현이 가능하긴 하지만 SSE 청크 단위 파싱에는 훨씬 번거롭고 불안정합니다.

③ fetch + ReadableStream이 표준 방식 — POST 기반 SSE 구현은 fetch → response.body.getReader() → reader.read() 루프가 현재 브라우저 표준입니다. conversation_created, stream_complete, error 등 커스텀 SSE 이벤트를 버퍼로 파싱하는 것도 이 방식으로만 가능합니다.
결정 전역 스토어 없이 Vue 3 Composition API의 ref / computed를 Composable 패턴으로 조합해 상태 관리 근거 ① useChatState 호출부가 ChatPage.vue 단 하나 — ref()는 호출마다 새 인스턴스를 생성하므로, 한 곳에서만 호출하고 자식 컴포넌트에 props로 내려주는 구조라면 전역 공유가 불필요합니다.

② 라우트 간 공유 상태 없음 — 각 페이지가 독립적으로 동작하며, 범용적인 useAuth · useTheme 정도만 App.vue에서 사용합니다. Pinia가 필요한 순간은 "두 컴포넌트가 공통 부모 없이 같은 상태를 공유해야 할 때"인데 그 상황이 발생하지 않았습니다.

③ onMounted 패턴과 Pinia의 구조적 불일치 — useChatState 내부에서 onMounted를 직접 호출하고 있습니다. Pinia 스토어는 컴포넌트 라이프사이클 훅을 사용할 수 없어 이 패턴 자체가 Pinia와 맞지 않습니다.
